1
Operator Prepares a Run
Dashboard
Configurations Tab

The operator selects a reusable configuration, reviews task intent, start URL, verify text, high-risk keywords, and runtime notes.

→
Sensitive Data Tab

Customer profile JSON is already stored as encrypted records with generated vault://... references.

→
Per-Run Profile Selection

The operator checks the specific customer profiles this operation is allowed to use. This becomes the run's explicit permission set.

2
Dashboard Launches the Job
Backend
Launch Request

The dashboard calls POST /api/configurations/:id/launch with runtime notes and selected sensitiveDataGrants.

→
Create Job
  • normalizes configuration details
  • persists runtime metadata
  • stores attached customer profile grants
  • returns the new job.id
→
agent_jobs + agent_steps

Job, goals, metadata, replay timeline, and future operator prompts are all anchored to persistent rows in Postgres.

3
Backend Spawns a Local Worker
Worker Manager
spawnWorkerForJob()

The backend-managed worker process model starts a Python worker bridge for the selected job instead of relying on BullMQ / Redis.

→
Platform Worker

The worker loads job input and metadata, including attached customer profiles, high-risk keywords, and any saved operator context.

→
Live Status Event

The dashboard receives job.created and job.updated over Socket.IO and auto-selects the new process.

4
LangGraph Plans and Starts Executing
observe -> act -> verify
Python Worker
Planner Context

The worker includes allowed domains, completion criteria, operator prompts, and attached customer profile labels in the planning prompt.

→
Execution Loop
  • observe current page state
  • choose next action
  • validate via backend policy
  • verify page change and goal status
→
Dashboard Updates

Process progress, replay events, and goal timeline updates stream back to the selected process view.

5
A Form Needs Customer-Specific Sensitive Data
Vault Flow
Runtime Resolution

The worker reaches a field that uses a vault://customer_a/ssn style reference and requests the backend to resolve it at execution time.

→
Grant Enforcement
  • checks the current job id
  • loads sensitiveDataGrants
  • blocks ungranted customer profiles
  • limits access to granted fields only
→
Encrypted Record

If allowed, the backend decrypts only the requested field from sensitive_records; raw PII never needs to live in the planning prompt.

6
High-Risk Action or Sensitive Reveal Requires Approval
Approval Inbox
Create Approval

The backend creates an approval request for risky browser actions or sensitive record access and marks the job as waiting when needed.

→
Approvals Tab

The operator sees the pending approval, policy reason, and target context inside the dashboard approval inbox.

→
Approve or Reject

The operator decides in real time. The worker retries the action or data request only after the approval is recorded.

7
Dashboard Audits the Run in Real Time
Monitor + Jobs
Replay Stream

Mutating actions produce screenshot-based replay cards. Non-mutating events appear in the background event list.

→
Step Timeline

The vertical goal tracker shows completed goals, current work, and clear failure reasons for unfinished steps.

→
Process Context

Recent operator prompts, screenshots, and action summaries stay attached to the selected process for auditing.

8
Operator Corrects or Extends the Run
Correct + Resume
Prompt Selected Process

The operator can send a new prompt, append steps, or use Correct + Resume to recover from failure.

→
Update Profile Grants

The operator can also attach a different customer profile to this specific process if the correction requires a new customer dataset.

→
Append Context + Resume Worker

The backend persists the new operator prompt and merged grants, then re-spawns or resumes the worker with the updated state.

9
Run Completes with Auditable State
Persisted Outcome
Worker Final State
  • completed, failed, paused, or cancelled
  • goal completion and failure reason
  • final response / result payload
→
Stored Audit Trail
  • agent_jobs metadata
  • agent_steps timeline
  • replay_chunks screenshots
  • attached customer-profile grants
→
Operator Review

The operator can now inspect the completed process, replay evidence, customer-profile usage, and any approvals that were required.
