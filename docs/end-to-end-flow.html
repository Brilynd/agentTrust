1
User Sends a Message
Chrome Extension
Chat Tab (popup.js)

User types a message in the Chat tab and presses Send. The extension calls POST /api/commands with the user's JWT token and the message payload.

→
Command Queue

Backend authenticates the user JWT, stores the command in the pending queue. The command waits for the agent to pick it up.

→
Agent Long-Poll

The Python agent is continuously long-polling GET /api/commands/pending with its M2M JWT. It receives the message instantly.

2
Identity Verification
Auth0
M2M Token Request

Agent exchanges client_id + client_secret for an Auth0 M2M JWT via POST /oauth/token. Token is cached until expiry.

→
Auth0 IdP
  • Validates client credentials
  • Issues RS256-signed JWT
  • Includes scopes: browser.basic, browser.form.submit
  • Publishes public keys via JWKS endpoint
→
JWKS Validation

Backend fetches Auth0's public signing keys from /.well-known/jwks.json, verifies the JWT signature (RS256), audience, issuer, and expiration. Keys cached 24 hours.

3
Prompt Storage & Session Binding
Backend + RDS
Store Prompt

Agent calls POST /api/prompts with the user message. Backend stores it in the prompts table linked to the active session. Returns a prompt_id.

→
PostgreSQL (RDS)
  • INSERT into prompts table
  • Linked to sessions.id
  • Progress column for live UI updates
→
CloudWatch Logs

Fire-and-forget PROMPT_CREATED event sent to the prompts log stream in the agentTrust log group for centralized monitoring.

4
Intent Classification
CLASSIFY node
GPT-4.1-nano
Keyword Pre-Check

Fast keyword scan for browser-requiring terms: search, navigate, amazon, weather, today, latest, etc. If matched, skip LLM — classify as BROWSER immediately.

→
GPT-4.1-nano

If no keyword match, sends a one-shot prompt to the nano model. Returns a single word: BROWSER or CHAT. Near-instant response.

→
Route Decision
  • BROWSER → Continue to Plan node
  • CHAT → Skip to respond (no browser)
5
Task Planning & Decomposition
PLAN node
GPT-4.1-mini
RAG Context Injection

Retrieves top-3 similar past tasks from the action history via TF-IDF cosine similarity. Injects proven action patterns into the planner prompt.

→
GPT-4.1-mini Planner
  • Decomposes request into 2-6 sub-goals
  • API-first: prefers call_external_api over browser
  • Comparison tasks forced into single goal
  • Returns JSON list of goals
→
Live Progress

Agent pushes "PLAN | Planning: [goal text]" to PATCH /api/prompts/:id. Extension polls and renders a live thinking breakdown in the Chat tab.

6
Page Observation
OBSERVE node — no LLM call
Chrome Browser
Selenium Capture
  • Current URL and page title
  • Truncated visible text (first 4000 chars)
  • Interactive elements (links, buttons, inputs with hrefs)
  • Tab state (active tab, list of open tabs)
→
State Assembly
  • Login state detection (checks if already signed in)
  • Overlay/popup detection
  • Screenshot capture (base64)
  • Loop detection (3+ identical scrolls triggers hint)
→
URL Rewriting

Programmatic guards: Google Images redirected to Web Search, login/sign-in URLs redirected to root, GitHub tabs rewritten to clean paths.

7
Action Selection
AGENT node
GPT-4.1
GPT-4.1 Reasoning

Given the observation, current sub-goal, and conversation history, the model picks exactly one tool call. Available tools:

  • open_link — navigate to URL
  • click_element — click by index
  • type_text — type into input
  • scroll_page — scroll up/down
  • go_back — browser back
  • switch_tab — change tab
  • auto_login — multi-step login
  • call_external_api — GitHub/Google API
  • done — goal complete
→
Tool Call Dispatch

The selected tool call is routed to the BrowserActionExecutor. Before execution, it must pass through AgentTrust validation (Step 8).

8
Policy Enforcement & Risk Classification
The trust boundary — no action bypasses this
Backend API
POST /api/actions

InterceptedWebDriver sends action details (type, URL, domain, target, form data) with the M2M JWT Bearer token.

→
Risk Classification Engine
  • Domain sensitivity (financial +2, unknown +1)
  • Action type (form submit +1)
  • Keyword matching (delete, transfer +3)
  • URL patterns (/admin, /settings +2)
  • Form field analysis (password +2)
→
Risk Decision

Score determines outcome:

  • LOW 0-1 pts — allowed
  • MED 2-3 pts — allowed, flagged
  • HIGH 4+ pts — step-up required
  • BLOCKED domain — flat denied
8b
Policy Decision Branches
ALLOWED
Backend returns 201 with action_id and riskLevel. Action proceeds to execution (Step 9). Logged as status: allowed.
STEP-UP REQUIRED
Backend returns 403 with approvalId. Extension shows approval banner. Agent long-polls GET /api/approvals/:id/wait for up to 60s. User approves or denies. If approved, agent retries with approvalId. Logged as approved_override.
DENIED
Blocked domain or policy violation. Backend returns 403 with no approval option. Agent receives denial, skips this action, and increments failure counter. Logged as status: denied.
9
Audit Logging & CloudWatch
Every action logged — allowed, denied, or step-up
RDS + CloudWatch
SHA-256 Hash Chain

Each action's hash depends on the previous: SHA256(prev_hash + agent_id + type + timestamp + domain + url + risk). Tamper-evident — modifying any record breaks the chain.

→
PostgreSQL INSERT
  • Full action record with hash
  • Agent identity (agent_id from JWT)
  • Session + prompt references
  • Risk level, status, reason
  • Screenshot attached via PATCH
→
AWS CloudWatch Logs
  • Fire-and-forget — never blocks execution
  • Log group: agentTrust
  • Streams: actions, sessions, prompts, users, credentials, routines, connections
  • Event types: ACTION_CREATED, SESSION_CREATED, etc.
  • Centralized monitoring across all record types
10
Browser Execution
Action approved — now it runs
Chrome (Intercepted)
BrowserActionExecutor
  • execute_navigation() — Selenium .get(url)
  • execute_click() — find element + .click()
  • execute_form_submit() — fill fields + submit
  • auto_login() — multi-step: username, continue, password, submit
→
Chrome Browser

Selenium executes the action in the real browser. Overlay dismissal runs first (cookie banners, popups, passkey dialogs suppressed via CDP). Page waits for document.readyState.

→
Screenshot Capture

After execution, a screenshot is captured and uploaded to the backend via PATCH /api/actions/:id. Attached to the audit record.

10b
External API Path (GitHub / Google Calendar)
API-first — bypasses the browser entirely
Auth0 Token Resolution
call_external_api

Agent sends provider, method, URL, and body to POST /api/external/call. Backend classifies API risk (GET=low, POST=med, DELETE=high).

→
Auth0 Management API
  • Extract sub claim from user JWT
  • GET /api/v2/users/{sub}
  • Read provider token from identities[]
  • Fallback: Token Vault exchange
→
Proxy to Provider

Backend makes the actual API call to api.github.com or googleapis.com using the provider access token. Response is sanitized (tokens/secrets stripped) before returning to the agent.

11
Action Verification
VERIFY node
Python Agent
Result Check
  • Did the page change? (URL or content)
  • Did the action return success?
  • Is the current sub-goal complete?
  • Was there an error or timeout?
→
Retry / Advance Logic
  • Success → advance to next sub-goal
  • Failure → retry (up to 3 per goal)
  • 3 failures → skip to next goal
  • Loop detected → break and move on
→
Progress Update

Emits progress line to extension: step status, current goal, action taken. Only emits VERIFY on failure to keep the log clean.

↺ Loop: Steps 6 → 7 → 8 → 9 → 10 → 11 repeat for each sub-goal until all goals complete or max actions reached
All Sub-Goals Completed (or max actions)
12
Response Generation
RESPOND node
GPT-4.1-mini
Response LLM

GPT-4.1-mini generates a user-facing summary of what was accomplished: pages visited, data found, actions taken, any failures. Concise and actionable.

→
Store Response

Agent calls PATCH /api/prompts/:id with the response text. Also pushes final "DONE" progress line. Backend updates the prompt record in PostgreSQL.

→
UI Render

Extension polls GET /api/sessions, detects the new response. Renders in the Chat tab: thinking block (collapsible), then the response bubble. DOM diffing prevents flicker.

13
Action History Storage (RAG)
Learning for future tasks
Python Agent
Record Task Pattern

If the task was successful, the full action sequence (tools, args, results) is saved to the local .data/action_history.jsonl file with the original user request. TF-IDF index is rebuilt. Future similar tasks will benefit from these proven patterns.

→
Session Telemetry

Session-level telemetry emitted to CloudWatch: total actions, duration, goals completed, failures. Enables dashboard monitoring and alerting across all agent sessions.
